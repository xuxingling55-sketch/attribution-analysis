## 一、线索业务

> 线索是电销业务的核心资产。用户通过不同渠道触达后，被系统或人工分配给坐席，形成线索记录。
>
> 不同线索来源的业务流程差异较大，但最终都会进入线索分配表（`dw.fact_clue_allocate_info`），被坐席领取后进入线索领取记录表（`aws.clue_info`）。

### 1.1 电话线索（系统线索）

> **线索来源**：`mid_school`（中学业务）
> **地位**：第一大线索来源
> **别称**：电话线索 = 系统线索，对应线索来源聚合口径中的"电话线索"

**业务流程：**

```
用户端内活跃
    ↓
触发指定推送条件
    ↓
数仓推送线索到电销系统
    ↓
进入公海池过滤规则
    ↓  ← 不满足条件的被拒绝（deny_reason / deny_index）
满足条件，进入公海池
    ↓
按学段分流（小学 / 初中 / 高中）
    ↓
等待坐席领取
    ↓
坐席领取 → 进入线索分配表 → 形成电话线索
```

**关键环节说明：**

| 环节 | 说明 | 关联数据 |
|------|------|---------|
| 用户活跃 | 用户在 App 内产生活跃行为 | `dws.topic_user_active_detail_day` |
| 推送 | 数仓根据规则将用户推送到电销系统 | `aws.crm_active_data_pool_day`.`push_u_user` |
| 公海池过滤 | 推送后经过过滤规则，不满足条件的被拒绝 | `dw.fact_telesale_clue_day`.`deny_reason` / `deny_index` |
| 进入公海池 | 通过过滤规则的用户进入公海池等待领取 | `aws.crm_active_data_pool_day`.`enter_datapool_u_user` |
| 学段分流 | 公海池内按用户学段（小学/初中/高中）分组 | — |
| 坐席领取 | 坐席从公海池主动领取线索 | `dw.fact_clue_allocate_info`、`aws.clue_info` |

**核心漏斗（月度汇总表 `aws.crm_active_data_pool_month`）：**

| 漏斗层级 | 字段 | 含义 |
|---------|------|------|
| 第1层 | `active_u_user` | 当月活跃用户 |
| 第2层 | `push_u_user` | 当月被推送到电销的用户 |
| 第3层 | `enter_datapool_u_user` | 当月进入公海池的用户 |
| 第4层 | `recieve_u_user` | 当月被坐席领取的用户（仅 `mid_school` 来源） |

→ 漏斗表字段详解、业务术语映射、属性取值时点等详见 `code/sql/表结构/aws.crm_active_data_pool_month.sql`

### 1.2 企微线索

> **线索来源**：`WeCom`（企业微信）
> **触达方式**：通过企业微信添加好友建立联系

**业务流程：**

```
用户在 App 内活跃
    ↓
在指定资源位曝光企微入口
    ↓
用户点击入口
    ↓
跳转到企微小程序，曝光坐席二维码
    ↓
用户点击/扫描坐席二维码，添加坐席微信
    ↓
自动拉取入库 → 形成企微线索
```

**与电话线索的关键区别：**

| 维度 | 电话线索 | 企微线索 |
|------|---------|---------|
| 触达方式 | 电话外呼 | 企业微信 |
| 线索产生 | 数仓推送 → 公海池 → 坐席领取 | 用户主动添加坐席微信 → 自动入库 |
| 是否经过公海池 | 是 | 否 |
| 线索来源字段值 | `mid_school` | `WeCom` |

### 1.3 定向分配线索

> **线索来源**：`mid_school_manual`（中学业务-人工）
> **产生方式**：由 BI 筛选一部分优质线索，定向分配给坐席

- 不经过公海池流程，直接分配
- 通过 `note` 字段区分不同批次的定向分配（如"定向分配公海池数据"、"定向分配付费数据"等）
- 转化率默认看领取**当月内**转化 → `glossary.md` #定向分配线索转化率

### 1.4 人工录入线索

> **线索来源**：`manual`（人工录入）
> **产生方式**：坐席自行将用户录入到系统中

- 转化率口径与定向分配一致（领取当月内转化），`note` 字段可按需区分具体线索批次

### 1.5 客服推送线索

> **线索来源**：`custom_service_manual`（客服推送）
> **产生方式**：用户咨询客服过程中表现出购买意向，客服将用户推送给电销

- 目前由团长平均承接这些线索
- 团长承接后通过**线索转移**分配给一线坐席进行转化
- 转化率口径：默认不限制领取人与转化人的对应关系（谁领取谁转化），其他条件与定向分配一致（领取当月内转化）

### 1.6 自助线索池

> **线索来源**：`self_service_pool`（自助线索池）
> **产生方式**：线索被坐席领取后，5 天内若未通过重新进入公海池或以其他方式重新入库，则自动放入自助公海池

- 自助公海池中的线索可被坐席自助领取

### 1.7 公海池

> 公海池是电话线索流程中的核心中间环节，是线索从"推送"到"被领取"之间的等待区。

- **定义**：数仓推送的线索经过过滤规则后，满足条件的进入公海池，等待坐席领取
- **分流逻辑**：进入公海池后按用户学段（小学/初中/高中）分组
- **过滤规则**：不满足条件的线索会被拒绝，拒绝原因记录在 `dw.fact_telesale_clue_day` 的 `deny_reason` / `deny_index` 字段
- **适用线索来源**：仅电话线索（`mid_school`）经过公海池流程
- **关联表**：
  - 日粒度：`aws.crm_active_data_pool_day`
  - 月粒度：`aws.crm_active_data_pool_month`

## 二、服务期业务

> 服务期是用户与业务团队之间的归属关系，决定了用户的营收归属。
> 底层数据记录在 `user_allocation.user_allocation`，团队名称关联 `user_allocation.team`。
> **数据范围**：仅 C 端用户，不包含 B 端用户。
>
> ⚠️ **唯一性待确认**：技术团队文档写有唯一约束（同一用户同一时间只允许一条有效服务期），但营收表 `topic_order_detail.team_names` 存在包含多个服务期的情况（服务期双算），具体机制待确认。

### 2.1 服务期生命周期概览

```
用户注册（有 uid）
    ↓
业务团队认领用户 → 创建服务期记录（state = effective）
    ↓
用户购买商品 → 触发服务期延长
    ↓                          ↗ abandon：主动放弃
服务期到期或提前终止 → 终止原因 → expired：自然过期
                               ↘ cancelAfterSales：退款后权益不足
    ↓
进入冷却期（原团队 90 天内不可再次认领）
    ↓
冷却期结束 → 可被重新认领
```

### 2.2 售前认领规则

**认领条件：**
- 首次接触用户时，把用户认领到自己团队
- 用户必须在洋葱系统中已注册（有 uid）

**认领冷却期（硬约束）：**
- 用户服务期到期后，原归属团队 **90 天内不允许**再次认领该用户
- 免冷却期的情况：
  - 销售侧人工接触用户后，手动录入线索认领
  - 用户在业务专属售卖渠道购买正价商品及大会员商品
  - 电销专属：认领 48 小时后若无有效通话主动放弃，可免冷却期再次认领

**自动重试认领间隔（软约束）：**
- 认领失败时，接口返回 `adoptRetryTime`（下次可重试时间点）
- 非人工触发业务至少保证 91 天重试间隔
- 历史线索建议设置为 91 天后重试

### 2.3 服务期延长规则

**触发条件：**
- 已认领用户购买团队关注的商品类型，且订单支付成功
- 订单用户必须为服务中用户

**延长计算方式：**

| 商品类型 | 延长方式 |
|---------|---------|
| 大会员商品 | 固定延长 36500 天（≈永久） |
| 正价商品（固定时间模式） | `now()` + 配置天数 |
| 正价商品（权益时间模式） | 权益截止时间 + 配置天数 |
| 非正价商品 | `now()` + 配置天数（暂无人使用） |

**特殊商品处理：**
- 研学商品：伪权益期限 = 支付时间 + 365 天
- 样机售卖：伪权益期限 = 支付时间 + 30 天
- 取最长伪课程权益和真课程权益的最大值

### 2.4 服务期缩减规则

**触发条件：**
- 已认领用户订单退款成功
- 订单用户必须为服务中用户

**缩减后果：**
- 可能导致服务期直接失效（如认领期刚过，退款唯一订单）

**特殊商品缩减：**
- 查询历史成交订单，计算伪权益期限
- 若用户非大会员，对比伪课程权益和真课程权益，取最大值

### 2.5 服务期终结规则

**终止原因（对应 `user_allocation.user_allocation.end_reason`）：**

| end_reason | 含义 | 冷却期 |
|-----------|------|-------|
| `abandon` | 主动放弃 | 免冷却期 |
| `expired` | 自然过期 | 90 天冷却期 |
| `cancelAfterSales` | 商品退款后权益不足 | 90 天冷却期 |

**自然过期机制：**
- 底层每分钟扫描到期用户，更新 `state` 为 `invalid`
- 业务层推荐在到期前 5 分钟主动终结，避免底层触发"自然过期"飞书报警

## 三、转介绍业务

> 转介绍是电销团队的裂变营销手段：在洋葱付费的老用户，邀请身边朋友来洋葱买课。
> 转介绍是线索来源之一（`clue_source = 'referral'`），但拥有独立的业务流程、归属规则和积分体系。
>
> 指标定义与计算口径 → `glossary.md` #十一、转介绍相关

### 3.1 完整业务流程

```
【推荐人资格判定】
    ├─ 条件a：在洋葱（B端除外）历史单笔付费 >= 108元
    └─ 条件b：兑换码订单，权益来源为兑换码兑换/批量发放奖励，权益状态有效，任意单科权益 >= 30天
    ↓
推荐人（老用户/坐席）下载专属海报        ← crm.promotion_poster
    ↓
推荐人分享海报到朋友圈/好友              ← crm.point_log_all (point_type=1)
    ↓
潜在用户看到海报/活动页                  ← events.frontend_event_orc (曝光事件)
    ↓
点击进入推荐有礼活动页                   ← events.frontend_event_orc (点击事件)
    ↓
被推荐人扫码进入转介绍小程序，授权手机号   ← authorization_status=1
    ↓
【系统判断：是否满足转介绍规则】
    │ 被推荐人条件：近60天未进入过电销库 & 无服务期归属或在电销服务期
    │
    ├─ ✅ 满足 → 绑定转介绍关系（crm.new_user, channel=2）
    │            → 线索入库来源 = 'referral'（aws.clue_info）
    │
    └─ ❌ 不满足 → 线索入库来源 = 'telesale_mp'/'server_number'/'parent'
                   → 不建立转介绍绑定关系
    ↓
新用户分配到坐席名下（线索归属规则见 3.3）
    ↓
坐席跟进转化 → 新用户付费                ← aws.crm_order_info
    ↓
推荐人获得积分奖励（按积分规则见 3.4）
```

**关键环节与关联数据：**

| 环节 | 说明 | 关联表 |
|------|------|--------|
| 下载海报 | 坐席在CRM下载，或用户在小程序自行下载 | `crm.promotion_poster` |
| 分享朋友圈 | 付费用户转发朋友圈，获得积分 | `crm.point_log_all`（`point_type=1`） |
| 活动页曝光/点击 | 推荐有礼活动页的前端埋点 | `events.frontend_event_orc` |
| 转介绍绑定 | 记录推荐人→被推荐人关系 | `crm.new_user`（`channel=2`） |
| 线索入库 | 被推荐人进入电销系统 | `aws.clue_info`（`clue_source='referral'`） |
| 付费转化 | 被推荐人购买课程 | `aws.crm_order_info` |

### 3.2 核心业务规则

| 规则 | 内容 |
|------|------|
| 推荐人资格 | 历史单笔付费 >= 108元（B端除外），或有效兑换码（单科权益 >= 30天） |
| 被推荐人条件 | 不限是否注册过，近60天未进电销库 & 无服务期归属或在电销服务期 |
| 推荐方式 | 被推荐人通过转介绍海报扫码进入小程序授权手机号 |
| 绑定时效 | 推荐人与被推荐人绑定关系有效期 **6个月**（精确到小时） |
| 线索在库有效期 | 被推荐人与坐席绑定 **30天**，未成单自动释放 |

### 3.3 线索归属规则

**满足转介绍规则的线索**（入库来源 = `referral`）：

| 场景 | 归属逻辑 |
|------|---------|
| 坐席A下载老用户B的海报 → A在职 | 新用户推送到坐席A名下 |
| 坐席A下载老用户B的海报 → A不在职 | 新用户推送到老用户B当前所属坐席名下 |
| 用户自行下载海报 | 按上月转介绍营收（CRM口径）>= 5000元的坐席依次分配 |

**不满足转介绍规则的线索**（入库来源 = `telesale_mp`/`server_number`/`parent`）：
- 通过海报进入的：归属逻辑同上
- 未通过海报进入转介绍小程序的：按上月转介绍营收 >= 5000元的坐席依次分配

**特殊场景：**
- 付费客户在A坐席名下且A下载了海报 → 该海报带来的所有裂变新客户归属A
- 付费客户掉库后，之前A下载的海报裂变新客户仍归属A；掉库后新下载的海报按新规则判归属

### 3.4 积分奖励规则

**朋友圈积分**：最高参与10次，总计5500积分（可兑换50元京东卡）

**邀请注册激活积分**（激活 = 任意看完1个洋葱视频）：

| 条件 | 积分 |
|------|------|
| 累计邀请1名新用户注册并激活 | 500 |
| 累计邀请3名新用户注册并激活 | 2000 |
| 累计邀请5名新用户注册并激活 | 2500 |

**购课积分**（大型活动如暑促采用双倍）：

| 单笔购课金额 | 积分 |
|-------------|------|
| 498 < 金额 <= 2000 | 5000 |
| 2000 < 金额 <= 3000 | 10000 |
| 3000 < 金额 <= 4000 | 15000 |
| 4000 < 金额 <= 5000 | 20000 |
| >= 5000 | 30000 |

**累计购课金额积分**（绑定后好友累计购课，含复购）：

| 累计金额 | 积分 |
|---------|------|
| > 5000 | 3000 |
| > 10000 | 5000 |
| > 20000 | 8000 |
| > 30000 | 10000 |
| > 40000 | 15000 |
| > 50000 | 20000 |

### 3.5 不满足转介绍条件的线索来源

通过转介绍海报/小程序进入但不满足转介绍绑定条件的用户，会以非转介绍来源入库：

| 线索来源 | 含义 |
|---------|------|
| `telesale_mp` | 电销小程序（洋葱学园+小程序） |
| `server_number` | 洋葱服务号 |
| `parent` | 家长端小程序 |

这些来源在 `aws.clue_info` 中记录，但不在 `crm.new_user`（`channel=2`）中建立转介绍绑定关系。
