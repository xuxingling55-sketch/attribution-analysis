# 数据知识库建设规范

> **目标**：将公司数据资产结构化，构建 AI 可读、人可维护的统一知识库。test
>
> **核心原则**：
> - **单一事实源**：业务指标定义只在 glossary 写一次，其他位置引用；表级元数据（字段说明、枚举值、筛选条件）写在各 DDL 文件中，跟随表走
> - **DDL 即文档**：表结构元数据直接写在 DDL 文件的表头注释和字段 COMMENT 中，不另建索引层
> - **结构化优先**：使用表格、枚举列表、代码块等结构化格式，减少自然语言叙述

## 三、信息归属规则

新增知识时，按以下规则确定放在哪个模块：

| 要记录的信息类型 | 放在哪里 |
|-----------------|---------|
| 指标定义、计算公式、适用表 | `glossary/【域】_维护人.md` |
| 同一指标在不同场景下的口径差异 | `glossary/【域】_维护人.md` |
| 多张表怎么选（选表指南） | `glossary/【域】_维护人.md` 或 `business-context/` 中对应域 |
| 一张表的粒度、字段说明、统计口径 | DDL 文件表头注释 |
| 查表时必须加的 WHERE 条件 | DDL 文件表头【常用筛选条件】 |
| 简单的单表关联方式 | DDL 文件表头【常用关联】 |
| 字段的枚举值和含义映射 | DDL 文件第三段（枚举值） |
| 复杂的多表 JOIN 场景、全局 ER 图 | `table-relations.md` |
| 业务流程、数据产生链路 | `business-context/【域】_维护人.md` |
| 高频取数场景的标准 SQL 模板 | `sql-patterns.md` |

## 四、各模块编写规范

### glossary/（业务知识字典）

按业务域**分文件**维护（文件名：`【域】_维护人.md`）；文件内仍按主题分章节（如用户分层、订单相关、线索相关等）。每个术语/指标按以下结构编写：

```markdown
### {指标名称}

- **定义**：{用一句话说清楚这个指标是什么}
- **计算方式**：{标准 SQL 表达式，如 COUNT(DISTINCT user_id)、SUM(amount)}
- **表来源**：{从哪张表取，多张表时列出每张表对应的字段差异}
- **筛选条件**：{如果和该表的常用筛选条件不一致，说明差异；一致则不用写}
- **⚠️ 注意**：{容易混淆的地方、口径差异、待验证信息等，没有则不写}
```

**填写规则**：

| 字段 | 是否必填 | 说明 |
|------|---------|------|
| 定义 | 必填 | 一句话，说清楚"是什么" |
| 计算方式 | 必填 | 给出可直接用的 SQL 表达式 |
| 表来源 | 必填 | 写明 `schema.table_name`；多张表可用时，列出差异 |
| 筛选条件 | 按需 | **仅当**与该表的常用筛选条件不同时才写，相同则省略 |
| ⚠️ 注意 | 按需 | 口径差异、易混淆、待验证内容；没有则不写 |

**示例 1：简单指标**

```markdown
### 领取线索量 / 消耗线索量

- **定义**：被领取的独立用户数
- **计算方式**：`COUNT(DISTINCT user_id)`
- **表来源**：`aws.clue_info`
- **⚠️ 注意**：同一用户可被多次领取，统计线索量时要去重
```

**示例 2：多表可选的指标**

```markdown
### 营收金额

- **定义**：订单实收金额汇总
- **计算方式**：
  | 表 | SQL 表达式 |
  |----|-----------|
  | 全量订单表 `dws.topic_order_detail` | `SUM(order_amount)`（订单去重后）或 `SUM(sub_amount)`（子订单粒度） |
  | 电销订单表 `aws.crm_order_info` | `SUM(amount)` |
- **表来源**：`dws.topic_order_detail`、`aws.crm_order_info`
- **筛选条件**：`status = '支付成功'`（比该表常用筛选多了 status 限制）
```

**示例 3：有口径差异的指标**

```markdown
### 转化用户量

- **定义**：有订单的用户数
- **计算方式**：
  | 表 | SQL 表达式 |
  |----|-----------|
  | 全量订单表 | `COUNT(DISTINCT u_user)` |
  | 电销订单表 | `COUNT(DISTINCT user_id)` |
- **表来源**：`dws.topic_order_detail`、`aws.crm_order_info`
- **筛选条件**：
  - 看营收时：加 `status = '支付成功'`
  - 看转化率时：不限 status
- **⚠️ 注意**：转化 vs 营收的 status 规则不同，详见本章节顶部说明
```

**其他约定**：

- 枚举值明细不在此处展开，引用对应的 DDL 文件
- 一组强关联指标（如漏斗的各层级）可用引用块 `>` 写一段共同说明，再逐个列出各指标
- 待验证内容统一用 `⚠️` 标记并说明原因

### table-relations.md（表间关联关系）

- 维护全局 ER 图，新增核心表时补充
- 每个关联包含：关联字段对照 + SQL 示例 + 注意事项
- 易错场景给出 ❌ 错误写法 和 ✅ 正确写法

### sql-patterns.md（标准 SQL 参考库）

- 模板必须可直接执行，参数用 `${param}` 占位
- 标注参考的原始 SQL 文件路径
- 口径对齐 `glossary/` 中对应域文件的定义

### business-context/（业务背景知识）

- 流程用 ASCII 图展示关键环节
- 每个环节标注关联的表和字段
- 不写计算口径（那是 glossary 的职责）

### DDL 文件（code/sql/表结构/）

DDL 文件是每张表的**唯一权威文档**，采用三段式结构：

```
┌─────────────────────────────┐
│  第一段：表头注释             │  这张表是什么、怎么用、有什么坑
├─────────────────────────────┤
│  第二段：DDL 建表语句         │  字段名、类型、COMMENT
├─────────────────────────────┤
│  第三段：枚举值               │  关键字段有哪些取值、每个值什么含义
└─────────────────────────────┘
```

---

#### 第一段：表头注释

```sql
-- =====================================================
-- {表中文名} {schema}.{table_name}
-- =====================================================
--
-- 【表粒度】★必填
--   {一条记录代表什么业务动作？用哪个字段唯一标识？}
--   {同一个用户/订单会出现多条吗？如果会，说明什么维度会导致多条}
--
-- 【业务定位】有同类易混表时写
--   {这张表的业务角色是什么？}
--   {和哪张表容易搞混？区别是什么？}
--   {什么场景该用本表 vs 该用另一张表？}
--
-- 【数据来源】汇总表/宽表/应用层表写
--   {数据从哪些上游表加工而来？}
--   {每个上游表提供了什么数据？}
--
-- 【统计口径】事实表写
--   {用这张表做核心指标时，标准的 COUNT / SUM 怎么写？}
--   {格式：指标名 = SQL表达式}
--   {注意：业务指标的权威定义在 glossary（本仓库为 glossary/【域】_维护人.md），本段仅记录"用本表怎么算"}
--
-- 【常用关联】需要频繁 JOIN 时写
--   {最常和哪张表 JOIN？用什么字段对什么字段？}
--   {关联时有什么注意事项？}
--
-- 【常用筛选条件】事实表写
--   ★必加条件（任何查询都必须带）：
--   {格式：- SQL条件  -- 为什么必须加}
--
--   场景条件（按分析目的选加）：
--   {格式：- SQL条件  -- 什么场景下加}
--
-- 【注意事项】有坑时写
--   {哪些字段已废弃？推荐用什么替代？}
--   {哪些字段容易用错？正确用法是什么？}
--   {有哪些口径陷阱？}
--
-- =====================================================
```

**规则**：
- 【表粒度】必填，其余段落按需添加
- 没有的段落直接不写，不留空段落
- 花括号 `{}` 内的文字是填写指引，实际写的时候替换成具体内容

**各段落填写示例**：

**【表粒度】**（必填）

```sql
-- 【表粒度】
--   一笔子订单 = 一条记录（order_id + sub_good_sk 唯一）
```

```sql
-- 【表粒度】
--   一个用户被领取一次 = 一条记录（info_uuid 唯一标识）
--   同一用户(user_id)可被同一或不同坐席(worker_id)领取多次
```

**【业务定位】**

```sql
-- 【业务定位】
--   仅电销业务订单，是电销营收分析的权威数据来源
--   与全公司订单宽表(dws.topic_order_detail)的区别：
--   - 本表：仅电销业务订单
--   - 全量表：全公司所有业务订单（电销、新媒体、入校、体验营等）
--   因双服务期归属差异，本表营收 ≠ 全量表筛选电销后的营收
--
--   选表原则：
--   - 电销营收分析 → 用本表
--   - 活跃转化分析 / 判断用户是否购买过某商品 → 用全公司订单宽表
```

**【数据来源】**

```sql
-- 【数据来源】
--   aws.crm_active_data_pool_day  —— 日表按月聚合，提供漏斗数据 + 用户属性
--   dw.fact_telesale_clue_day     —— 公海池拒绝原因（deny_reason / deny_index）
--   dw.fact_clue_allocate_info    —— 坐席领取信息（含线索来源 clue_source）
```

**【统计口径】** — 仅记录"用本表怎么算"，业务指标的权威定义以 `glossary/` 对应域文件为准

```sql
-- 【统计口径】
--   外呼次数 = COUNT(action_id)
--   外呼线索量 = COUNT(DISTINCT user_id)
--   接通量 = COUNT(CASE WHEN is_connect = 1 THEN action_id END)
--   有效接通量 = COUNT(CASE WHEN is_valid_connect = 1 THEN action_id END)
```

**【常用关联】**

```sql
-- 【常用关联】
--   线索来源追溯：
--     本表.recent_info_uuid = aws.clue_info.info_uuid
--     · recent_info_uuid：成交前该坐席最近一次领取的线索id
--     · modify_recent_info_uuid：矫正后（排除人工录入来源），推荐优先使用
--
--   组织架构名称：
--     本表.regiment_id = dw.dim_crm_organization.org_id（获取团名称）
```

**【常用筛选条件】**

```sql
-- 【常用筛选条件】
--   ★必加条件：
--   - is_test = false               -- 排除测试订单，任何查询都要加
--   - worker_id <> 0                -- 排除无坐席记录
--
--   场景条件：
--   - status = '支付成功'           -- 看营收时加；看转化量时不加
--   - in_salary = 1                 -- 计入工资口径时加
--   - workplace_id IN (4, 400, 702) -- 限定电销职场时加
--   - regiment_id NOT IN (0, 303, 546) -- 排除特定团组时加
```

**【注意事项】**

```sql
-- 【注意事项】
--   ⚠️ 本表自带转化字段（paid_cnt, paid_amount 等）口径不明确，不建议使用
--      转化需求请直接关联 aws.crm_order_info 计算
--   ⚠️ good_type 已弃用，推荐使用 good_kind_name_level_2
--   ⚠️ tag 字段仅在 clue_source = 'mid_school' 时有效
```

---

#### 第二段：DDL 建表语句

每个字段必须有 COMMENT（实在找不到先空着），COMMENT 编写要求：

| 场景 | 写法 | 示例 |
|------|------|------|
| 简单枚举（≤5个值） | 直接写在 COMMENT 里 | `'是否接通 0否 1是'` |
| 复杂枚举（>5个值） | COMMENT 写字段含义，详见第三段 | `'线索来源（详见文件末尾枚举值）'` |
| 有推荐/废弃 | 标注使用建议 | `'商品类型(已弃用,推荐使用 good_kind_name_level_2)'` |
| 常被用于 WHERE | 标注常用筛选值 | `'销售职场id（常用筛选：4、400、702为电销职场）'` |

---

#### 第三段：枚举值

在 DDL 语句之后，用 SQL 注释列出该表关键字段的完整枚举值。

**格式**：

```sql
-- =====================================================
-- 枚举值
-- =====================================================
--
-- ## field_name（字段中文名）
--
-- > 补充说明（按需，如适用场景、与其他字段的关系、注意事项等）
--
-- | 枚举值 | 含义 |
-- |--------|------|
-- | value1 | 说明 |
-- | value2 | 说明 |
```

**什么字段需要列枚举**：

| 情况 | 处理方式 |
|------|---------|
| 取值可穷举、对写 SQL 有影响 | 完整列出 |
| 取值多但有常用子集 | 列出常用值，注明"部分枚举" |
| 布尔/二值字段（0/1、true/false） | 直接写在 COMMENT 里，不用到第三段 |
| 自由文本（name、备注等） | 不列 |
| 已下线/废弃的枚举值 | 保留但标注 🔴 |

**完整示例**（外呼记录表第三段）：

```sql
-- =====================================================
-- 枚举值
-- =====================================================
--
-- ## call_state（拨打状态）
--
-- > ⚠️ 默认不使用。判断是否接通统一用 is_connect 字段。
-- > 特殊需求明确要用时，用 call_status（由 call_state 派生），不用 call_state。
--
-- | 枚举值 | 含义 |
-- |--------|------|
-- | dealing | 已接通 |
-- | notDeal | 未接通 |
-- | abnormal | 异常 |
-- | blackList | 黑名单 |
-- | leak | IVR放弃 |
--
-- ## channel_id（外呼渠道）
--
-- | 枚举值 | 渠道名称 |
-- |--------|---------|
-- | 1 | 七陌 |
-- | 2 | 天眼 |
-- | 34 | 蘑谷云 |
-- | 38 | 百川 |
-- | 39 | 百悟 |
-- | 101 | 智鱼 |
-- | 102 | 天田 |
```

### Cursor Rule（.cursor/rules/）

Cursor Rule 是编写 SQL 时自动注入的规则，用于强制执行语法标准和口径约定。

**文件格式**：`.mdc` 文件，包含 YAML Front Matter 和正文

```yaml
---
description: {一句话描述规则的适用场景}
globs: "**/*.sql"           # 匹配哪些文件时生效
alwaysApply: true           # 是否始终生效
---
```

**编写原则**：

| 原则 | 说明 |
|------|------|
| 一个规则解决一类问题 | 不要把 SQL 语法标准和业务口径混在同一个文件 |
| 规则可执行 | 写"禁止 `select *`"而不是"尽量避免" |
| 给出正例 | 每条规则附上符合规范的代码示例 |
| 避免与知识库重复 | 口径定义放 `glossary/` 域文件，Rule 只引用或约束写法 |

## 五、跨模块引用规范

模块间通过统一语法交叉引用，避免重复定义：

| 引用目标 | 语法 |
|---------|------|
| 术语定义 | `→ glossary/【域】_维护人.md#{术语名}`（本仓库按域拆分，见 `glossary/README.md`） |
| 表结构/枚举值/筛选条件 | `→ code/sql/表结构/【域】_维护人/{文件名}.sql` |
| 关联关系 | `→ table-relations.md#{场景}` |
| SQL 模板 | `→ sql-patterns.md#{模板编号}` |
| 业务流程 | `→ business-context/【域】_维护人.md#{流程名}` |

**原则**：定义只写一次，其他位置用引用语法指向权威模块。

> 单文件版知识库仍可使用 `glossary.md`、`business-context.md` 等命名；与本仓库 `cursor2.0` 的域拆分写法等价，二选一保持团队一致即可。

## 六、维护流程

### 新增内容时的标准动作
