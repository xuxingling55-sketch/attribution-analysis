# 洋葱学园异动归因分析框架 v1.0

> **适用场景**：当核心经营指标出现异动时，按本框架自动生成假设、用数据验证、定位根因
> **使用者**：业务方（运营/策略同事）
> **数据源**：Impala（SSH 隧道）
> **最后更新**：2026-04-02

---

## 一、框架总览

### 1.1 核心流程

```
业务方输入异动描述
    ↓
Step 1: 异动解析 — 提取指标、维度、时间范围、变化方向
    ↓
Step 2: 假设生成 — 按分析树 + LLM 动态补充，生成假设列表
    ↓
Step 3: 数据验证 — 对每个假设生成 SQL，连 Impala 查询
    ↓
Step 4: 根因排序 — 按贡献度排序，标注主因/次因
    ↓
Step 5: 报告输出 — 结构化报告 + 可视化
```

### 1.2 适用的异动类型

| 类型 | 示例 |
|------|------|
| **转化率异动** | "上周9年级线索转化率跌了" |
| **营收/GMV异动** | "昨天GMV比上周同期下降15%" |
| **客单价异动** | "本月小学客单价下降了" |
| **流量异动** | "初中活跃用户连续3天下降" |
| **电销效率异动** | "本周电销接通率异常偏低" |

## 二、异动解析模块 (Step 1)

### 2.1 输入解析

从业务方的自然语言描述中提取结构化要素：

| 要素 | 说明 | 示例 |
|------|------|------|
| **指标** | 哪个指标异动 | 转化率、GMV、客单价、接通率 |
| **维度** | 哪个切面 | 9年级、小学、电销渠道、新增用户 |
| **时间** | 什么时间段 | 上周、昨天、本月、近7天 |
| **方向** | 升还是降 | 下跌、上涨、异常 |
| **幅度** | 变化多少（可选） | 下降15%、减少200人 |

### 2.2 基线确定

自动选择对比基线：

| 异动时间 | 默认基线 | 备选基线 |
|----------|----------|----------|
| 昨天 | 上周同天 | 前天 |
| 上周 | 上上周 | 去年同期 |
| 本月 | 上月 | 去年同月 |
| 活动期 | 去年同活动 | 活动前一周 |

## 三、假设生成模块 (Step 2)

### 3.1 通用分析树（6大维度 × 子假设）

基于历史复盘文档和业务经验沉淀的完整假设树：

### 维度一：流量/线索结构变化

> 核心逻辑：分母端的量和质发生变化，导致指标波动

| 假设编号 | 假设描述 | 验证指标 | 数据来源 |
|----------|----------|----------|----------|
| H1.1 | 线索总量变化（领取量/消耗量增减） | 线索领取量、消耗线索量（去重user_id） | `aws.clue_info` |
| H1.2 | 线索来源结构变化（公海池/企微/人工录入占比） | 各 `clue_source` 占比变化 | `aws.clue_info` |
| H1.3 | 活跃用户规模变化 | 活跃人数（C端必筛条件） | `dws.topic_user_active_detail_day` |
| H1.4 | 渠道流量质量变化（如华为/OPPO等渠道质量下降） | 分渠道活跃量和转化率 | `dws.topic_user_active_detail_day` |
| H1.5 | 线索质量变化（接通率、有效通话率下降） | 接通率、有效接通占比 | `aws.clue_info` + 通话记录表 |

### 维度二：用户结构变化

> 核心逻辑：不同分层用户的转化率和客单差异巨大，结构变化可直接影响大盘指标

| 假设编号 | 假设描述 | 验证指标 | 数据来源 |
|----------|----------|----------|----------|
| H2.1 | 用户分层占比变化（新增/老未/续费/定金/历史大会员/历史组合品） | 各层占比 & 各层转化率 | `business_user_pay_status_business` 字段 |
| H2.2 | 低转化用户涌入（新增用户占比上升拉低大盘） | 新增用户占比变化 & 转化率 | `dws.topic_user_active_detail_day` |
| H2.3 | "被洗过"的老用户占比升高（对促销疲态） | 历史参加过大促的老用户占比 | 订单历史 + 活跃表交叉 |
| H2.4 | 年级/学段分布变化 | 各学段活跃人数和转化率（`mid_stage_name`, `mid_grade`） | 活跃表/订单表 |
| H2.5 | 身份分布变化（家长 vs 学生，注意：用 `real_identity`，禁用 `role`） | 家长/学生占比及各自转化率 | `dw.dim_user` |

### 维度三：销售执行问题

> 核心逻辑：电销是重要转化渠道，人效和执行质量直接影响转化

| 假设编号 | 假设描述 | 验证指标 | 数据来源 |
|----------|----------|----------|----------|
| H3.1 | 外呼量/接通率下降 | 外呼次数、接通率、有效通话率 | 电销行为数据表 |
| H3.2 | 团队/职场维度异常（如新职场 vs 成熟职场差异） | 分团队转化率、分职场人效 | `dw.dim_crm_organization` |
| H3.3 | 个人维度异常（个别 top sales 请假或状态异常） | 个人维度转化率排名变化 | 电销表 + 订单表 |
| H3.4 | 跟进时效下降（线索从领取到首次触达的时间延长） | 平均跟进时长 | `aws.clue_info` 时间字段 |
| H3.5 | 新人占比升高拉低人效（新设/扩招导致平均能力下降） | 新人/老人占比 & 各自转化率 | 组织架构表 |

### 维度四：商品/价格策略因素

> 核心逻辑：商品结构和定价策略直接影响客单价和转化率，且对不同渠道影响差异大

| 假设编号 | 假设描述 | 验证指标 | 数据来源 |
|----------|----------|----------|----------|
| H4.1 | 在售商品结构变化（单科/组合品/大会员占比） | 各商品类型订单数和金额占比 | `dws.topic_order_detail` |
| H4.2 | 客单价分布变化（低价商品占比上升拉低 AOV） | 价格区间分布 & 各区间订单占比 | 订单表 |
| H4.3 | 优惠/折扣力度变化（促销策略调整） | 优惠金额、折扣率 | 订单表优惠字段 |
| H4.4 | 定金策略影响（定金门槛变化影响漏斗上层） | 定金量、定金尾款转化率 | 订单表 |
| H4.5 | 商品与渠道的适配性错配（高客单商品依赖电销，APP 自主转化受损） | 分渠道 × 分商品类型的转化率 | 订单表 |
| H4.6 | 售卖模式调整（如组合升单加购 vs 常规模式） | 调整前后的人效、当配营收占比 | 订单表 |

### 维度五：外部/时间因素

> 核心逻辑：教育行业有强周期性，考试/放假节奏直接影响需求进场时间

| 假设编号 | 假设描述 | 验证指标 | 数据来源 |
|----------|----------|----------|----------|
| H5.1 | 考试周影响（期中/期末/模考期间活跃和转化下降） | 异动时段是否处于考试周 | 校历/外部日历 |
| H5.2 | 节假日/放假时间影响（寒暑假启动延迟导致需求进场推迟） | 同比活跃曲线的位移 | 活跃表同比 |
| H5.3 | 开学季/返校影响 | 活跃用户结构的周期性变化 | 活跃表 |
| H5.4 | 竞品活动干扰 | 异动时段是否有竞品大促/营销活动 | 外部情报 |
| H5.5 | 同环比基线异常（去年同期本身就是异常高/低点） | 对比基线的合理性检查 | 历史数据 |

### 维度六：产品/功能因素

> 核心逻辑：APP 端的改版、资源位调整可能影响曝光和转化漏斗

| 假设编号 | 假设描述 | 验证指标 | 数据来源 |
|----------|----------|----------|----------|
| H6.1 | APP 改版/功能上线影响（如"世界观改版"对售卖曝光的负面干扰） | 改版前后的曝光量、点击率、转化率 | 埋点数据 |
| H6.2 | 资源位调整（学习页/我的Tab/宝藏页的曝光变化） | 分场景营收占比变化 | 埋点数据 |
| H6.3 | 申诉/退款异常（申诉比例提升导致净营收流失） | 申诉量、退款金额 | 订单表退款字段 |

---

### 3.2 假设筛选逻辑

不是所有假设都需要验证，根据异动类型自动筛选相关假设：

| 异动指标 | 优先验证的维度 | 次优先 |
|----------|---------------|--------|
| **转化率** | 维度一(流量) + 维度二(用户结构) + 维度三(销售执行) | 维度四(商品) + 维度五(外部) |
| **GMV/营收** | 全维度（营收 = 流量 × 转化率 × 客单价，三因子分解） | — |
| **客单价** | 维度四(商品价格) + 维度二(用户结构) | 维度一(流量质量) |
| **活跃/流量** | 维度一(流量) + 维度五(外部) + 维度六(产品) | — |
| **电销效率** | 维度三(销售执行) + 维度一(线索) | 维度二(用户结构) |

### 3.3 LLM 动态补充

在固定分析树之外，LLM 根据以下信息动态补充假设：
- 异动描述中的特殊关键词（如"9年级"→ 检查中考政策变化）
- 近期已知的业务动作（如刚调整了定金策略、刚上线了新商品）
- 历史同类异动的根因模式

## 四、数据验证模块 (Step 3)

### 4.1 验证流程

```
对每个假设：
1. 生成验证 SQL（遵循知识库的口径规范）
2. 通过 SSH 隧道连 Impala 执行
3. 计算变化幅度和贡献度
4. 判断假设成立/不成立/部分成立
```

### 4.2 SQL 生成规则（必须遵循）

| 规则 | 说明 |
|------|------|
| **C端活跃必筛** | `product_id = '01' AND client_os IN ('android','ios','harmony') AND active_user_attribution IN ('中学用户','小学用户','c')` |
| **排除测试** | 宽表 `is_test_user = 0`；电销表 `is_test = false` |
| **转化率口径** | 计算转化量/率时**不加** `status = '支付成功'`；计算营收时**必须加** |
| **正价订单** | 实收金额 ≥ 39 元 |
| **手机号解密** | `phone` 字段使用 `unbase64` 解码 |
| **身份判断** | 用 `real_identity`，**禁用** `role` |
| **学段/年级** | 用 `mid_stage_name`（修正学段）和 `mid_grade`（修正年级） |
| **电销营收** | 电销专题必须使用 `aws.crm_order_info`，**不用**宽表的 `business_gmv_attribution = '电销'` |

### 4.3 贡献度计算

对于营收异动，使用三因子分解：

```
营收 = 流量 × 转化率 × 客单价

Δ营收 = Δ流量贡献 + Δ转化率贡献 + Δ客单价贡献

其中：
Δ流量贡献 = (流量_t - 流量_t-1) × 转化率_t-1 × 客单价_t-1
Δ转化率贡献 = 流量_t × (转化率_t - 转化率_t-1) × 客单价_t-1
Δ客单价贡献 = 流量_t × 转化率_t × (客单价_t - 客单价_t-1)
```

对于转化率异动，使用结构分解：

```
大盘转化率 = Σ(分层占比_i × 分层转化率_i)

拆解为：
- 结构效应：各分层占比变化对大盘的影响
- 效率效应：各分层自身转化率变化对大盘的影响
```

## 五、根因排序模块 (Step 4)

### 5.1 排序标准

| 排序维度 | 权重 | 说明 |
|----------|------|------|
| **贡献度** | 40% | 该因素能解释多少比例的异动 |
| **统计显著性** | 30% | 变化是否超出正常波动范围（±2σ） |
| **可行动性** | 20% | 是否有对应的改善措施 |
| **时间吻合度** | 10% | 变化发生的时间是否与异动时间匹配 |

### 5.2 输出分级

| 级别 | 标准 | 标签 |
|------|------|------|
| **主因** | 贡献度 > 50% 且统计显著 | 🔴 |
| **次因** | 贡献度 10%-50% | 🟡 |
| **弱相关** | 贡献度 < 10% 或不显著 | ⚪ |

## 七、实战示例：上周9年级线索的转化和客单都跌了

### Step 1: 异动解析
- **指标**：转化率 + 客单价（双指标异动）
- **维度**：9年级
- **时间**：上周 vs 上上周
- **方向**：双跌

### Step 2: 自动筛选假设
因为是转化率+客单价双跌，优先验证：

**转化率方向：**
- H1.1 9年级线索量变化
- H1.2 9年级线索来源结构变化
- H2.1 9年级内用户分层占比变化
- H2.3 "被洗过"的老用户占比升高
- H3.1 负责9年级的电销团队外呼量/接通率
- H3.2 团队/职场维度异常
- H5.1 是否处于中考相关考试周

**客单价方向：**
- H4.1 9年级在售商品结构变化
- H4.2 低价商品占比上升
- H4.3 优惠力度加大
- H2.1 低客单用户群（如新增用户）付费占比上升

### Step 3-5: 执行验证 → 排序 → 报告

（连接 Impala 后自动执行）
