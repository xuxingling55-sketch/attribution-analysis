## 1. 框架概览

归因分析遵循"六问框架"，每一步有明确的输入、判定逻辑和输出：

```
Q1 异动精确化  →  明确指标 / 对比基准 / 异动准确性校准
        ↓
Q2 真伪判定    →  数据质检，排除采集/口径问题
        ↓
Q3 定位下钻    →  全量 vs 局部判断 → 多维对比 → 专项深挖
        ↓
Q4 假设验证    →  只验证显著维度，已排除项直接标注
        ↓
Q5 主因判定    →  三因子拆解量化贡献度
        ↓
Q6 建议输出    →  金字塔结构，取数给建议
```

**核心原则**：
- Q2 不通过 → 立即终止，修复数据后重跑
- Q3 先判断全量/局部，再决定下钻方向
- Q6 每条建议必须可执行，禁止虚指

## 2. Q1 异动精确化

### 2.1 输入规范

| 参数 | 说明 |
|---|---|
| 指标 | GMV / 付费用户数 / 线索量 / 转化率 / ARPU |
| 对比基准 | 根据场景选择（见下方） |
| 异动标准 | 与业务方提前对齐（见下方） |

### 2.2 对比基准选择

**没有固定的"正确"基准，根据业务场景判断：**

| 场景 | 推荐基准 | 原因 |
|---|---|---|
| 日常波动监控 | **近四周同星期均值** | 消除周内节奏差异；教育电商周末天然高峰，不应视为离群 |
| 有明显季节性 | **同比去年同周** | 升学季、寒暑假等强季节性指标，环比失真 |
| 紧急排查 | **环比上周** | 快速定位，适合突发异动，但需注意周内节奏差异 |
| 大促后复盘 | **去促基线** | 大促周本身不可比，需剔除大促影响后重建基线 |

> ⚠️ **选定基准后不能中途换**。若分析过程中发现基准选错（如基线周含大促），应回到 Q1 重新选基准，而不是在 Q3 中修正。

### 2.3 异动准确性校准

**归因开始前，必须与业务方对齐以下两点：**

**① 多大幅度算异常？**

不使用固定百分比，而是与历史波动范围对比：
- 如果该指标历史周波动本来就在 ±10%，那 -8% 不算异常
- 如果历史波动在 ±3%，那 -5% 就值得归因

可参考近四周同星期的标准差作为参照系，但最终标准由业务方确认。

**② 是否有已知业务原因？**

归因前先排除已知原因，不浪费分析资源：
- 已知：本周有课程大规模到期 → 正常退潮，记录原因，不进归因
- 已知：本周竞品搞大促 → 标注外部因素，酌情归因
- 未知：无任何已知业务原因 → 进入正式归因流程

### 2.4 本案示例

```
指标：全学段 C 端 GMV
周期：2026-04-13 vs 2026-04-06
对比基准：近四周同星期均值（日常监控场景）

GMV 当前周：4,456,178 元
近四周同星期均值：4,681,322 元
变化：-4.8%

业务方确认：无已知大促/活动，-4.8% 超出历史正常波动范围
→ 触发归因流程
```

## 3. Q2 真伪判定

**核心原则：涉及哪张表才检查哪张，不是每次都查所有表。**

### 3.1 通用质检（所有涉及的表都必须过）

| 质检项 | 方法 | 判定标准 |
|---|---|---|
| **可比性（Gate）** | 两周是否含节假日/大促节点不对等 | 不对等 → 切换基准或标注"非可比周"，**不往后走** |
| **分区完整性** | 当前周每日行数 vs 近四周同星期均值 | 偏差 > 50% 且无业务原因 → 标记[数据延迟]，**终止归因** |
| **口径一致性** | 两周过滤条件完全相同 | `product_id` / `is_test_user` / `sub_amount` 等必须一致 |

> ⚠️ **可比性是最优先的 Gate**：两周节奏不对等，其他质检没有意义。

### 3.2 各表专项质检

#### topic_order_detail（C端GMV / 付费用户）
- 分区完整性：日 `order_cnt` vs 近四周同星期均值
- 基线合理性：基线周是否含大促波峰？若含 → 切"去促基线"
- ★ **不做极值剔除**（电商单笔金额分布稳定）

#### crm_order_info（电销GMV / 电销转化）
- 分区完整性：日 `order_cnt`
- **P99 极值剔除**：`amount > PERCENTILE(amount, 0.99)` 的单子过滤
  - 电销单笔金额差异大，超大单会扭曲趋势
- 日期字段用 `pay_time`（不是 `paid_time`！）

#### topic_user_active_detail_day（活跃用户 / CVR / 漏斗）
- 分区完整性：日 UV
- 归因字段校验：`business_gmv_attribution` 分布是否突变
- 确认事件类型完整（show / click / pay 等均有数据）

#### clue_info（线索量 / 电销触达率）
- 分区完整性：日 `clue_cnt`
- `source_type` 分布校验：线索来源结构是否突变
  - 结构突变可能是录入问题，不是真实业务变化

## 4. Q3 定位下钻

Q3 分三层，**核心逻辑是先判断问题性质，再决定下钻方向**：

```
第一步：全量 vs 局部判断
        ↓
第二步：多维对比（局部问题专用）
        ↓
第三步：专项深挖（锁定具体跌点后）
```

### 4.1 第一步：全量 vs 局部判断

**这是 Q3 最重要的一步，决定后续所有分析方向。**

| 判断结果 | 现象 | 分析方向 |
|---|---|---|
| **全量问题** | 所有维度同向下跌，无明显分化 | 大概率系统性原因：季节性退潮 / 大促后回落 / 外部环境。重点看时段分布和同比，不急于下钻具体维度 |
| **局部问题** | 部分维度下跌，部分正常甚至上涨 | 有结构性问题，进入第二步多维对比，锁定异动维度 |

**判断方法**：拉一张"各维度变化率热力表"，快速扫描是否存在分化：

```sql
-- 以学段为例，快速看各维度是否同向
SELECT
    stage_name,
    SUM(CASE WHEN paid_time >= '{BASE_START}' AND paid_time < '{BASE_END}'
             THEN sub_amount ELSE 0 END) AS base_gmv,
    SUM(CASE WHEN paid_time >= '{CURR_START}' AND paid_time < '{CURR_END}'
             THEN sub_amount ELSE 0 END) AS curr_gmv,
    (curr_gmv - base_gmv) / base_gmv * 100 AS pct_change
FROM dws.topic_order_detail
WHERE paid_time >= '{BASE_START}' AND paid_time < '{CURR_END}'
  AND product_id = '01'
  AND sub_amount >= 39
  AND is_test_user = 0
GROUP BY 1
ORDER BY pct_change ASC
```

> **判断技巧**：如果最高变化率和最低变化率差距 < 5%，基本是全量问题；差距 > 10% 且有维度逆势上涨，基本是局部问题。

### 4.2 第二步：多维对比（局部问题）

**不是找"谁贡献最大"，而是用对比定性"哪里有个性问题"：**

#### 对比维度 A：年级 × 渠道

最常用的交叉，定位"哪个学段 × 哪个渠道"是跌点：

| 发现 | 定性 |
|---|---|
| 初中所有渠道都跌 | 初中学段有结构性问题（季节/产品/竞争） |
| 所有学段商业化渠道都跌 | 商业化投放效率问题（非学段问题） |
| 初中商业化跌，初中电销涨 | 渠道间有分化，商业化特有问题 |

**SQL 模板**：
```sql
SELECT
    stage_name,
    business_gmv_attribution AS channel,
    SUM(CASE WHEN paid_time >= '{BASE_START}' AND paid_time < '{BASE_END}'
             THEN sub_amount ELSE 0 END) AS base_gmv,
    SUM(CASE WHEN paid_time >= '{CURR_START}' AND paid_time < '{CURR_END}'
             THEN sub_amount ELSE 0 END) AS curr_gmv
FROM dws.topic_order_detail
WHERE paid_time >= '{BASE_START}' AND paid_time < '{CURR_END}'
  AND product_id = '01'
  AND sub_amount >= 39
  AND is_test_user = 0
GROUP BY 1, 2
HAVING base_gmv > 10000
ORDER BY (curr_gmv - base_gmv) ASC
```

#### 对比维度 B：用户分层

判断是哪类用户在流失，定性获客/留存/复购哪个环节出问题：

| 用户层 | 字段 | 发现示例 |
|---|---|---|
| 新用户首购 | `real_identity = '新用户'` | 新用户跌 → 获客/转化链路问题 |
| 老用户续费 | `business_user_pay_status_business` | 续费跌 → 产品/服务质量问题 |
| 高净值用户 | 付费金额 TOP 20% | 高净值跌 → 高价课/套餐策略问题 |
| 流失召回用户 | 历史付费但近期无活跃 | 召回跌 → 私域/促销策略问题 |

#### 对比维度 C：公私域

判断流量来源结构是否在变化：

| 渠道类型 | 说明 | 发现示例 |
|---|---|---|
| 公域（APP自然流量） | 搜索/推荐/榜单 | 公域跌 → 平台算法/SEO问题 |
| 私域（推送/社群） | 微信/短信/企微 | 私域跌 → 运营/消息触达问题 |
| 付费投放 | 商业化/信息流 | 投放跌 → ROI/创意/落地页问题 |

#### 对比维度 D：渠道之间

横向对比各渠道表现，判断是渠道共性问题还是个性问题：

| 发现 | 定性 |
|---|---|
| 电销涨 + APP跌 | APP 端有问题，电销不受影响 |
| 所有渠道都跌 | 产品/外部原因，非渠道问题 |
| 只有商业化电商跌 | 电商平台（天猫/京东）特有问题 |

### 4.3 第三步：专项深挖

**在多维对比锁定具体跌点后，才进入专项深挖。**

#### 专项 A：学段 → SKU 下钻

适用：已确认某学段有个性问题

```sql
SELECT
    stage_name,
    good_name,
    mid_grade,
    SUM(CASE WHEN paid_time >= '{BASE_START}' AND paid_time < '{BASE_END}'
             THEN sub_amount ELSE 0 END) AS base_gmv,
    SUM(CASE WHEN paid_time >= '{CURR_START}' AND paid_time < '{CURR_END}'
             THEN sub_amount ELSE 0 END) AS curr_gmv
FROM dws.topic_order_detail
WHERE stage_name IN ('{TOP_STAGE}')
  AND product_id = '01'
  AND sub_amount >= 39
  AND is_test_user = 0
GROUP BY 1, 2, 3
HAVING base_gmv > 5000
ORDER BY (curr_gmv - base_gmv) ASC
LIMIT 12
```

锁定 TOP 下跌 SKU 后，判断原因类型：

| 原因类型 | 验证方法 |
|---|---|
| 季节性退潮 | 对比去年同期同 SKU |
| 曝光骤降 | 查 `show` 事件 UV |
| 价格调整 | 对比两周 `sub_amount` 均值 |
| 内容质量 | 查完课率 / 差评率 |

#### 专项 B-1：APP 渠道 → 四层漏斗

适用：已确认 APP/商业化渠道有个性问题

```
曝光(show UV) → 点击(click UV) → 试听(trial UV) → 支付(pay UV)
```

逐层计算转化率，定位断点层级，并补充 iOS vs Android 分平台对比。

#### 专项 B-2：电销渠道 → 团队归因三步

适用：已确认电销渠道有个性问题

| 步骤 | 分析内容 | 表 |
|---|---|---|
| Step 1 | 按团队 × 学段拆 GMV | `crm_order_info`（P99过滤） |
| Step 2 | 线索量 vs 转化率拆分 | `clue_info` + `crm_order_info` |
| Step 3 | 团队人效对比 | 同上 |

## 5. Q4 假设验证

**⚡ 只验证 Q3 多维对比中显著的维度，不全跑六个假设。**

标准六假设供参考，实际按 Q3 结论选择验证项：

| 假设 | 验证指标 | 常用表 |
|---|---|---|
| H1 用户规模萎缩 | `COUNT(DISTINCT u_user)` | `topic_order_detail` |
| H2 商品结构变化 | `good_name` 维度变化 | `topic_order_detail` |
| H3 渠道效率下滑 | `business_gmv_attribution` | 订单表 + `crm_order_info` |
| H4 价格/AOV 变化 | `SUM(sub_amount)/COUNT(DISTINCT u_user)` | `topic_order_detail` |
| H5 外部竞品/政策 | 竞品监控、行业新闻 | 外部信息 |
| H6 APP平台/技术 | 漏斗转化率、支付成功率 | 埋点事件表 |

**判定结果记录格式**：

| 假设 | 验证结果 | 贡献度 | 判定 |
|---|---|---|---|
| H1 用户规模萎缩 | 付费用户 -8.4% | ~55% | 🔴 核心主因 |
| H4 价格变化 | AOV -0.2%，不显著 | ~0% | ✅ 已排除 |
| H6 APP技术 | iOS 漏斗数据待补充 | 待核 | 🟡 待确认 |

## 6. Q5 主因判定

### 三因子拆解

```
GMV = 付费用户数 × AOV
```

**量化各因子贡献**：
```
ΔGMV_用户 = ΔUser × BaseAOV
ΔGMV_AOV  = ΔAOV  × CurrUser
```

**本案示例**：

| 因子 | 变化 | ΔGMV贡献 | 贡献度 |
|---|---|---|---|
| 付费用户数 | 5,342 → 4,891（-8.4%） | -45.4万 | 176% |
| AOV | 877 → 910（+3.8%） | +4.4万 | -76%（对冲） |
| **合计** | | **-25.2万** | **100%** |

→ **用户规模萎缩是核心主因，AOV 反涨部分对冲**

## 7. Q6 建议输出

### 7.1 报告结构（金字塔原理）

```
① 背景（Background）
   指标 / 周期 / 对比基准 / 异动幅度

② 核心结论（Conclusion）← 最先看
   全量/局部判断 + 主因（贡献度%）+ 数据佐证

③ 业务建议（Action）← 可直接执行
   [数据事实] + [发现问题] + [具体行动]

④ 分析推导（Steps）← 最后看
   Q2质检 → Q3全量/局部判断 → 多维对比 → 专项 → Q4验证明细
```

### 7.2 建议输出规范

❌ **错误示范**：
```
"建议关注升学课的销售情况..."
"建议排查商业化渠道问题..."
```

✅ **正确示范**：
```
[小升初] GMV ↓24.5%（-3.1万），属局部问题（其他学段正常），
4月初签约高峰结束后自然退潮，建议4月底启动专题活动提前拉需求。

[商业化电商] ↓33.4%，其余渠道均正常，判断为渠道个性问题，
建议本周内核查商品链接状态与投放计划。
```

**格式**：`[数据事实] + [全量/局部定性] + [发现问题] + [具体行动]`

## 8. SQL 口径规范

### 8.1 核心表

| 表名 | 用途 |
|---|---|
| `dws.topic_order_detail` | C端订单，GMV / 付费用户主表 |
| `dws.topic_user_active_detail_day` | 用户活跃宽表，CVR / 漏斗分析 |
| `aws.crm_order_info` | 电销订单，电销GMV / 转化 |
| `aws.clue_info` | 线索表，电销触达率 |

### 8.2 必填过滤条件

**topic_order_detail**：
```sql
product_id = '01'               -- C端产品
is_test_user = 0                -- 排除测试用户
sub_amount >= 39                -- 正价单
paid_time >= 'YYYY-MM-DD'       -- timestamp 格式
```

**crm_order_info**：
```sql
is_test = false                 -- 排除测试订单
amount >= 39                    -- 正价单
amount <= [P99值]               -- P99 极值过滤
pay_time >= 'YYYY-MM-DD'        -- 注意是 pay_time！
```

### 8.3 字段使用规范

| 正确 ✅ | 禁止 ❌ | 说明 |
|---|---|---|
| `mid_grade` | `grade` | 年级字段 |
| `real_identity` | `role` | 用户身份字段 |
| `paid_time`（timestamp） | `day`（int分区） | 订单表日期过滤 |
| `pay_time` | `paid_time` | 电销表日期字段 |
| `good_name + stage_name` | `product_name` | 商品名 |
| `business_gmv_attribution` | `sales_team` | 渠道/电销归因字段 |

### 8.4 常用计算公式

```sql
-- GMV
SUM(sub_amount)

-- 付费用户数
COUNT(DISTINCT u_user)

-- AOV（客单价）
SUM(sub_amount) / COUNT(DISTINCT u_user)

-- 贡献度
(curr_gmv - base_gmv) / {total_delta} * 100

-- P99 阈值
SELECT PERCENTILE(amount, 0.99) FROM aws.crm_order_info WHERE ...

-- 近四周同星期均值（以周一为例）
SELECT AVG(daily_gmv)
FROM daily_stats
WHERE DAYOFWEEK(dt) = 2   -- 1=周日, 2=周一, ..., 7=周六
  AND dt >= DATE_SUB(CURRENT_DATE, INTERVAL 28 DAY)
```

## 9. 常见坑点

### 9.1 数据层

| 坑点 | 现象 | 处理方式 |
|---|---|---|
| 电销大单污染 | 电销 GMV 波动异常大 | 必须 P99 过滤 |
| 基线含大促 | 对比失真，误判为跌 | 切"去促基线"或同比 |
| 两周节奏不同 | 一周含周末高峰多，一周少 | 切近四周同星期均值 |
| 日期字段混用 | SQL 跑错数 | 订单表用 `paid_time`，电销用 `pay_time` |

### 9.2 分析层

| 坑点 | 现象 | 处理方式 |
|---|---|---|
| 跳过全量/局部判断直接下钻 | 把季节性退潮归因为渠道问题 | Q3 第一步必须先判断问题性质 |
| 只看贡献度不看对比 | 误判主因，忽视逆势维度 | 用对比定性，不只看贡献度排名 |
| 用户数和 AOV 分开看 | 漏掉对冲效应 | 必须做三因子拆解 |
| 电销逆势当成整体好转 | 掩盖 APP 端问题 | 电销和 APP 分开分析 |

### 9.3 报告层

| 坑点 | 现象 | 处理方式 |
|---|---|---|
| 没有标注全量/局部 | 读者无法判断严重程度 | 结论必须先定性 |
| 结论写在最后 | 读者看不到重点 | 金字塔结构，结论先行 |
| 建议不可执行 | "建议关注..." | 必须包含具体数字和行动 |

## 10. 实战案例

### 案例：2026-04-13 周 全学段 GMV 异动归因

**Q1**：GMV -4.8%，对比基准为近四周同星期均值，超出历史正常波动范围，业务方确认无已知原因，触发归因。

**Q2**：订单表日均偏差 4.4%，两周节奏一致，数据真实。电销表 P99 = 5,798 元，已过滤。

**Q3-第一步（全量 vs 局部）**：

初中 -7.1%，小学 -2.5%，高中 -1.8%；商业化 -14.3%，电销 +3.7%。  
→ **局部问题**（电销逆势上涨，存在明显维度分化），进入多维对比。

**Q3-第二步（多维对比）**：

| 对比维度 | 发现 | 定性 |
|---|---|---|
| 年级 × 渠道 | 初中×商业化（-14.9%），初中×商业化电商（-33.4%）是核心跌点 | 初中商业化有个性问题 |
| 用户分层 | 未付费老用户流失占 60% | 老用户复购/唤醒问题 |
| 渠道之间 | 电销 +3.7% vs 商业化 -14.3% | 商业化投放效率问题，非产品问题 |

**Q3-第三步（专项）**：

- SKU 下钻：[七升八][小升初][四升五] 升学规划课集中下跌，属季节性退潮
- 电销团队：初中电销 +3.7%，逆势亮点

**Q5**：付费用户贡献 176%，AOV 反涨对冲 76%，用户规模萎缩是唯一主因。

**Q6 建议**：

1. **[升学规划课 · 局部问题]** [七升八][小升初] GMV 分别下跌 7.9%/24.5%，属 4 月初签约高峰后自然退潮（对比去年同期同样下跌），建议 4 月底提前启动专题活动拉动下一波需求。

2. **[商业化电商 · 局部问题]** ↓33.4%，其余渠道均正常，判断为商业化电商个性问题，建议本周内核查商品链接状态、投放出价及落地页转化率。

3. **[未付费老用户 · 结构性流失]** 流失贡献 60%，建议结合私域数据分析老用户最后活跃时间节点，针对"课程到期未续费"用户启动定向召回活动。
