## 1. 框架概览

归因分析遵循"六问框架"，每一步有明确的输入、判定逻辑和输出：

```
Q1 异动精确化  →  明确指标 / 周期 / 统计显著性
        ↓
Q2 真伪判定    →  数据质检，排除采集/口径问题
        ↓
Q3 定位下钻    →  4维并行 → 交叉矩阵 → 专项深挖
        ↓
Q4 假设验证    →  只验证显著维度，已排除项直接标注
        ↓
Q5 主因判定    →  三因子拆解量化贡献度
        ↓
Q6 建议输出    →  金字塔结构，取数给建议
```

**核心原则**：
- Q2 不通过 → 立即终止，修复数据后重跑
- Q3 优先交叉，不全跑假设
- Q6 每条建议必须可执行，禁止虚指

## 2. Q1 异动精确化

### 2.1 输入规范

| 参数 | 说明 |
|---|---|
| 指标 | GMV / 付费用户数 / 线索量 / 转化率 / ARPU |
| 周期 | Current 周 vs Baseline 周（默认环比上周） |
| 参考窗口 | 过去 28 天历史数据 |
| 特殊情况 | 节假日周期可选同比，需在报告中标注 |

### 2.2 统计判定阈值（μ ± σ）

**不使用固定百分比阈值**，改用基于历史分布的统计方法：

**Step 1**：取过去 28 天每日指标值

**Step 2**：自动剔除离群天（迭代一次）
```
初始计算 μ₀, σ₀
标记 |x - μ₀| > 2σ₀ 的天 → 自动视为大促/节假日污染天
从样本中剔除，通常剔掉 3~5 天
```

**Step 3**：用剩余天重新计算最终 μ 和 σ

**Step 4**：将当前周日均代入判定
```
轻度异常：|当前周日均 - μ| > 2σ  →  触发 Q2 核查
严重异常：|当前周日均 - μ| > 3σ  →  必须立即归因
```

> **为什么用日数据算σ、用周日均判定？**  
> 日数据 28 个点，样本充足，σ 估计稳定；  
> 用周日均而非单日，平滑了周内偶发噪音（如某天系统故障），避免误触发。

> **为什么窗口选 28 天而非 56 天？**  
> 28 天 = 4 个完整周，覆盖一个月的周内节奏（同星期几都能比到），且季节性污染最小；  
> 56 天跨两个完整月，月初/月末的结构性差异会撑大 σ，导致异常难触发。

### 2.3 本案示例

```
指标：全学段 C 端 GMV
周期：2026-04-13 vs 2026-04-06

历史 28 天（剔除 2 个大促天后）：
  μ_daily = 668,760 元
  σ_daily = 41,230 元

当前周日均 = 622,311 元
|622,311 - 668,760| = 46,449 ≈ 1.13σ  →  轻度异常，触发 Q2

付费用户当前周日均 = 699 人
|699 - μ_user| > 3σ_user              →  严重异常，付费用户为核心归因指标
```

## 3. Q2 真伪判定

**核心原则：涉及哪张表才检查哪张，不是每次都查所有表。**

### 3.1 通用质检（所有涉及的表都必须过）

| 质检项 | 方法 | 判定标准 |
|---|---|---|
| **分区完整性** | 当前周每日行数 vs 过去 28 天日均 | 偏差 > 50% 且无业务原因 → 标记[数据延迟]，**终止归因** |
| **节奏可比性** | 两周是否含节假日/大促节点不对等 | 若有 → 切换同比基线或标注"非可比周" |
| **口径一致性** | 两周过滤条件完全相同 | `product_id` / `is_test_user` / `sub_amount` 等必须一致 |

> ⚠️ **可比性是 Gate**：节奏不对等直接切基线，不往后走。

### 3.2 各表专项质检

#### topic_order_detail（C端GMV / 付费用户）
- 分区完整性：日 `order_cnt` vs 28 天均值
- 基线合理性：基线周是否含大促波峰？若含 → 切"去促基线"
- ★ **不做极值剔除**（电商单笔金额分布稳定）

#### crm_order_info（电销GMV / 电销转化）
- 分区完整性：日 `order_cnt`
- **P99 极值剔除**：`amount > PERCENTILE(amount, 0.99)` 的单子过滤掉
  - 电销单笔金额差异大，超大单会扭曲趋势
- 日期字段用 `pay_time`（不是 `paid_time`！）

#### topic_user_active_detail_day（活跃用户 / CVR / 漏斗）
- 分区完整性：日 UV
- 归因字段校验：`business_gmv_attribution` 分布是否突变
- 确认事件类型完整（show / click / pay 等均有数据）

#### clue_info（线索量 / 电销触达率）
- 分区完整性：日 `clue_cnt`
- `source_type` 分布校验：线索来源结构是否突变
  - 结构突变可能是录入问题，不是真实业务变化

## 4. Q3 定位下钻

Q3 分三层，每层有明确的输入输出：

### 4.1 第一层：4 维并行下钻

**同时运行以下 4 个维度，互不依赖，不要串行：**

| 维度 | 字段 | 表 | 目的 |
|---|---|---|---|
| **A. 学段** | `stage_name` | `topic_order_detail` | 小学/初中/高中各自贡献度 |
| **B. 渠道** | `business_gmv_attribution` | 订单表 + `crm_order_info` | APP/电商/电销各自贡献度 |
| **C. 平台** | `client_os` | `topic_order_detail` | iOS/Android/PC 各自贡献度 |
| **D. 用户身份** | `real_identity` | `topic_order_detail` | 新用户/老用户/付费层级贡献度 |

**贡献度计算公式**：
```
贡献度 = ΔGMV_维度 / ΔTotal_GMV × 100%
```

> **注意**：渠道维度需要两张表分开查：  
> - APP/电商：`topic_order_detail`  
> - 电销：`crm_order_info`（需 P99 过滤）  
> 两张表结果合并展示。

**SQL 模板（学段示例）**：
```sql
SELECT
    stage_name,
    SUM(CASE WHEN paid_time >= '{BASE_START}' AND paid_time < '{BASE_END}'
             THEN sub_amount ELSE 0 END) AS base_gmv,
    SUM(CASE WHEN paid_time >= '{CURR_START}' AND paid_time < '{CURR_END}'
             THEN sub_amount ELSE 0 END) AS curr_gmv
FROM dws.topic_order_detail
WHERE paid_time >= '{BASE_START}' AND paid_time < '{CURR_END}'
  AND product_id = '01'
  AND sub_amount >= 39
  AND is_test_user = 0
  AND stage_name IS NOT NULL
GROUP BY 1
ORDER BY (curr_gmv - base_gmv) ASC
```

### 4.2 第二层：交叉矩阵

**条件触发**：第一层中贡献度绝对值最大的 1~2 个维度，进入交叉。

默认做"**学段 × 渠道**"交叉（这是最高频的组合），如果平台或用户身份维度也有显著信号（单维度贡献 > 20%），补做对应交叉。

**目的**：定位到"哪个学段 × 哪个渠道"才是真实跌点，避免两个维度互相掩盖。

**SQL 模板**：
```sql
SELECT
    stage_name,
    business_gmv_attribution AS channel,
    SUM(CASE WHEN paid_time >= '{BASE_START}' AND paid_time < '{BASE_END}'
             THEN sub_amount ELSE 0 END) AS base_gmv,
    SUM(CASE WHEN paid_time >= '{CURR_START}' AND paid_time < '{CURR_END}'
             THEN sub_amount ELSE 0 END) AS curr_gmv
FROM dws.topic_order_detail
WHERE paid_time >= '{BASE_START}' AND paid_time < '{CURR_END}'
  AND product_id = '01'
  AND sub_amount >= 39
  AND is_test_user = 0
  AND stage_name IS NOT NULL
  AND business_gmv_attribution IS NOT NULL
GROUP BY 1, 2
HAVING base_gmv > 10000
ORDER BY (curr_gmv - base_gmv) ASC
```

### 4.3 第三层：专项深挖

**根据交叉结果决定方向，只钻显著维度：**

#### 专项 A：商品/学段 → SKU 下钻

适用条件：学段维度贡献度 TOP，且交叉已锁定具体学段

```sql
SELECT
    stage_name,
    good_name,
    mid_grade,
    SUM(...) AS base_gmv,
    SUM(...) AS curr_gmv
FROM dws.topic_order_detail
WHERE stage_name IN ('初中', '小学')  -- 动态替换为 TOP 下跌学段
  AND product_id = '01'
  AND sub_amount >= 39
  AND is_test_user = 0
GROUP BY 1, 2, 3
HAVING base_gmv > 5000
ORDER BY (curr_gmv - base_gmv) ASC
LIMIT 12
```

锁定 TOP 下跌 SKU 后，判断原因类型：

| 原因类型 | 验证方法 |
|---|---|
| 季节性退潮 | 对比去年同期同 SKU，若同样下跌 → 自然周期 |
| 曝光骤降 | 查 `show` 事件 UV 是否骤降（商品下架/减曝光） |
| 价格调整 | 对比两周 `sub_amount` 均值，确认是否调价 |
| 内容质量 | 查完课率/差评率是否异常 |

#### 专项 B-1：APP 渠道 → 四层漏斗

适用条件：商业化/APP 渠道贡献度 TOP

```
曝光(show UV) → 点击(click UV) → 试听(trial UV) → 支付(pay UV)
```

逐层计算转化率，定位断点：
- 曝光量骤降 → 流量入口问题/商品下架
- 点击率下滑 → 商品卡片吸引力不足
- 试听率下滑 → 详情页/课程质量问题
- 支付率下滑 → 定价/活动/促销断档

**补充**：对 iOS vs Android 分平台做漏斗，定位单平台异常（发版/故障）。

#### 专项 B-2：电销渠道 → 团队归因三步

适用条件：电销渠道贡献度显著

| 步骤 | 分析内容 | 表 |
|---|---|---|
| Step 1 | 按团队 × 学段拆 GMV | `crm_order_info`（P99过滤） |
| Step 2 | 拆线索量 vs 转化率 | `clue_info` + `crm_order_info` |
| Step 3 | 对比团队人效（人均GMV） | 同上 |

- 线索少 → 获客问题（市场/投放）
- 线索多但转化低 → 销售话术/商品匹配问题
- 人效下滑 → 排查激励/培训/人员变动

## 5. Q4 假设验证

**⚡ 只验证 Q3 中显著的维度，不全跑六个假设。**

标准六假设供参考：

| 假设 | 验证指标 | 常用表 |
|---|---|---|
| H1 用户规模萎缩 | `COUNT(DISTINCT u_user)` | `topic_order_detail` |
| H2 商品结构变化 | `good_name` 维度贡献度 | `topic_order_detail` |
| H3 渠道效率下滑 | `business_gmv_attribution` | 订单表 + `crm_order_info` |
| H4 价格/AOV 变化 | `SUM(sub_amount)/COUNT(DISTINCT u_user)` | `topic_order_detail` |
| H5 外部竞品/政策 | 竞品监控、行业新闻 | 外部信息 |
| H6 APP平台/技术 | 漏斗转化率、支付成功率 | 埋点事件表 |

**判定结果记录格式**：

| 假设 | 验证结果 | 贡献度 | 判定 |
|---|---|---|---|
| H1 用户规模萎缩 | 付费用户 -8.4%，>3σ | ~55% | 🔴 核心主因 |
| H4 价格变化 | AOV -0.2%，不显著 | ~0% | ✅ 已排除 |
| H6 APP技术 | iOS 漏斗数据待补充 | 待核 | 🟡 待确认 |

## 6. Q5 主因判定

### 三因子拆解

```
GMV = 付费用户数 × AOV
```

**量化各因子贡献**：
```
ΔGMV_用户 = ΔUser × BaseAOV
ΔGMV_AOV  = ΔAOV  × CurrUser
ΔGMV_总   = ΔGMV_用户 + ΔGMV_AOV（两者之和 ≈ 实际总Δ）
```

**贡献度**：
```
用户贡献% = ΔGMV_用户 / ΔGMV_总 × 100%
AOV 贡献% = ΔGMV_AOV  / ΔGMV_总 × 100%
```

**本案示例**：

| 因子 | 变化 | ΔGMV贡献 | 贡献度 |
|---|---|---|---|
| 付费用户数 | 5,342 → 4,891（-8.4%） | -45.4万 | 176% |
| AOV | 877 → 910（+3.8%） | +4.4万 | -76%（对冲） |
| **合计** | | **-25.2万** | **100%** |

→ **用户规模萎缩是核心主因，AOV 反涨部分对冲**

## 8. SQL 口径规范

### 8.1 核心表

| 表名 | 用途 |
|---|---|
| `dws.topic_order_detail` | C端订单，GMV / 付费用户主表 |
| `dws.topic_user_active_detail_day` | 用户活跃宽表，CVR / 漏斗分析 |
| `aws.crm_order_info` | 电销订单，电销GMV / 转化 |
| `aws.clue_info` | 线索表，电销触达率 |

### 8.2 必填过滤条件

**topic_order_detail**：
```sql
product_id = '01'      -- C端产品，必须
is_test_user = 0       -- 排除测试用户
sub_amount >= 39       -- 正价单，排除异常低价
paid_time >= 'YYYY-MM-DD'  -- timestamp 格式
```

**crm_order_info**：
```sql
is_test = false        -- 排除测试订单
amount >= 39           -- 正价单
amount <= [P99值]      -- P99 极值过滤
pay_time >= 'YYYY-MM-DD'   -- 注意是 pay_time！
```

### 8.3 字段使用规范

| 正确 ✅ | 禁止 ❌ | 说明 |
|---|---|---|
| `mid_grade` | `grade` | 修正年级字段 |
| `real_identity` | `role` | 用户身份字段 |
| `paid_time`（timestamp） | `day`（int分区） | 订单表日期过滤 |
| `pay_time` | `paid_time` | 电销表日期字段 |
| `good_name + stage_name` | `product_name` | 商品名 |
| `business_gmv_attribution` | `sales_team` | 渠道/电销归因字段 |

### 8.4 常用计算公式

```sql
-- GMV
SUM(sub_amount)

-- 付费用户数
COUNT(DISTINCT u_user)

-- AOV（客单价）
SUM(sub_amount) / COUNT(DISTINCT u_user)

-- 贡献度
(curr_gmv - base_gmv) / {total_delta} * 100

-- P99 阈值
SELECT PERCENTILE(amount, 0.99) FROM aws.crm_order_info WHERE ...
```

## 9. 常见坑点

### 9.1 数据层

| 坑点 | 现象 | 处理方式 |
|---|---|---|
| 电销大单污染 | 电销 GMV 波动异常大 | 必须 P99 过滤 |
| 大促天污染 σ | 异常难触发 | 自动迭代剔除离群天 |
| 基线含大促 | 对比失真，误判为跌 | 切"去促基线"或同比 |
| 日期字段混用 | SQL 跑错数 | 订单表用 `paid_time`，电销用 `pay_time` |
| crm 表 stage 字段 | 字段名需核实 | 确认 `crm_order_info` 的学段字段名 |

### 9.2 分析层

| 坑点 | 现象 | 处理方式 |
|---|---|---|
| 只看总 GMV 不下钻 | 掩盖结构性变化 | 至少做学段 × 渠道交叉 |
| 用户数和 AOV 分开看 | 漏掉对冲效应 | 必须做三因子拆解 |
| 电销逆势当成整体好转 | 掩盖 APP 端问题 | 电销和 APP 分开分析 |
| top_stage 为空 | 脚本报错 | 加空值保护，默认取前两名 |

### 9.3 报告层

| 坑点 | 现象 | 处理方式 |
|---|---|---|
| 结论写在最后 | 读者看不到重点 | 金字塔结构，结论先行 |
| 建议不可执行 | "建议关注..." | 必须包含具体数字和行动 |
| 没有标注 σ 级别 | 无法判断严重程度 | 每个结论标注 >2σ 或 >3σ |

## 10. 实战案例

### 案例：2026-04-13 周 全学段 GMV 异动归因

**Q1 结论**：GMV 当前周日均 622,311，|x-μ| = 1.13σ（轻度异常）；付费用户 >3σ（严重异常），以付费用户为核心归因指标。

**Q2 结论**：订单表日均偏差 4.4%，两周节奏一致；电销表 P99 = 5,798 元，数据真实。

**Q3 结论**：

| 层次 | 发现 |
|---|---|
| 第一层（学段） | 初中贡献 77%（-17.3万），小学 19% |
| 第一层（渠道） | 商业化贡献 47%，商业化电商 14%；电销逆势 +3.7% |
| 第一层（平台） | iOS 贡献跌幅 44%，需补充漏斗数据 |
| 第一层（用户） | 未付费老用户流失占 60% |
| 第二层（交叉） | 初中×商业化（37%）、初中×商业化电商（14%）是核心跌点 |
| 第三层（SKU） | [七升八][小升初][四升五] 升学规划课集中下跌 |
| 第三层（电销） | 电销初中团队 +3.7%，属逆势亮点 |

**Q5 结论**：用户数贡献 176%，AOV 反涨对冲 76%，用户规模萎缩是唯一主因。

**Q6 建议**：
1. **[升学规划课]** GMV ↓24.5%（>2σ），属 4 月初签约高峰后自然退潮，建议 4 月底提前启动小升初/七升八专题活动拉动需求。
2. **[商业化电商]** ↓33.4%（>3σ 严重异常），建议本周内核查商品链接状态、投放计划及落地页转化，确认是否有下架或出价问题。
3. **[iOS 漏斗]** 贡献跌幅 44%，需补充 iOS 端埋点漏斗数据，排查近期是否有版本发布导致的功能异常。
